# AGENTS.md

Этот файл — первый вход для любого агента, который работает с репозиторием.

Он нужен не для замены канонических docs, а для того, чтобы агент:
- не расползался по scope
- не запускал бесконечные аудиты
- не превращал каждую задачу в рефактор всей подсистемы
- понимал, когда надо остановиться

## Важное правило

Этот файл **не переопределяет** канонические документы в `docs/`.
Если есть конфликт — см. `docs/00_governance/DOCUMENT_PRECEDENCE.md`.

## Порядок чтения

### Для любой задачи
1. `temp/AGENTS.md`
2. `temp/TASK_BRIEF.md`
3. один профильный канонический документ по теме задачи
4. только потом — конкретные файлы из разрешённого списка

### Для задач по world / mining / topology / reveal / presentation
Обязательный профильный документ:
- `docs/02_system_specs/world/DATA_CONTRACTS.md`

### Для задач по world generation
Обязательные профильные документы:
- `docs/04_execution/world_generation_rollout.md`
- `docs/04_execution/world_generation_gap_closure_plan.md`

## Что запрещено делать по умолчанию

Если это **не написано явно** в `temp/TASK_BRIEF.md`, агенту запрещено:
- чинить соседние найденные проблемы
- делать попутный рефактор
- запускать широкий архитектурный re-audit
- запускать perf-audit только потому, что «вдруг там что-то ещё не так»
- открывать или менять файлы вне allowlist
- расширять задачу до следующей итерации
- предлагать «идеальную финальную архитектуру» вместо закрытия текущего шага

Найденное вне scope уходит в блок **Out-of-scope observations**, а не в код.

## Что считается блокером

Задача считается незавершённой, если возникло хотя бы одно из условий:
- ломается smoke test из task brief
- появляется crash / assert / очевидная регрессия в затронутом пути
- нарушен публичный контракт или owner boundary
- в код попал новый hardcoded gameplay data или user-facing text
- task brief требует perf-ограничение, а оно явно не выполнено
- save/load ломается в затронутом пути

## Что НЕ считается блокером по умолчанию

Само по себе **не** является причиной продолжать работу бесконечно:
- «можно сделать API красивее»
- «можно ещё почистить архитектуру рядом»
- «есть ещё потенциальный refactor»
- «я увидел соседний contract gap»
- «внутреннюю реализацию можно сделать элегантнее»
- «можно гипотетически выиграть ещё немного времени, но текущий task этого не требует»

Если acceptance tests пройдены, а блокеров нет — задача должна останавливаться.

## Рабочая модель агента

На одну задачу допускается:
- 1 кодящий агент
- максимум 1 ревьюер

Не допускается:
- reviewer of reviewer
- audit chain из нескольких агентов на каждую мелкую правку
- запуск нескольких explore/review агентов без явной причины

## Правило минимального чтения

Агент не должен начинать с чтения половины репозитория.
Он должен сначала понять:
- цель шага
- stop condition
- allowed files
- relevant contract

Только после этого можно открывать код.

## Правило минимального изменения

Если задачу можно закрыть локальным изменением в существующем entrypoint, надо делать именно так.

По умолчанию агент не создаёт:
- новую подсистему
- новый manager
- новый service boundary
- новый runtime pipeline

Пока task brief прямо не требует этого.

## Формат результата после каждой задачи

Агент обязан вернуть:

### In scope completed
- что реально сделано

### Files changed
- какие файлы изменены

### Acceptance checks
- список проверок из task brief: passed / failed

### Out-of-scope observations
- что замечено, но не трогалось

### Smallest next step
- следующий минимальный шаг, а не новый большой эпос

## Когда звать arch/perf review

### Архитектурное ревью нужно только если:
- меняется public boundary
- меняется owner слоя данных
- добавляется новый mutation path
- задача явно про архитектурное решение

### Perf review нужно только если:
- задача затрагивает hot path
- уже есть измеренный overrun
- task brief требует конкретный perf contract

Во всех остальных случаях сначала делается узкая реализация по brief.

## Главный принцип

У этого проекта уже достаточно governance, чтобы держать порядок.

Чего не хватает — это дисциплины остановки.

Агент должен делать **ровно текущий шаг**, а не доводить всю подсистему до абстрактного совершенства.
