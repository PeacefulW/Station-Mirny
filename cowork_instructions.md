# Рекомендации для Cowork: Instructions и Scheduled Tasks

## Часть 1: Обзор проекта (Ревью)

### Общая оценка

**Станция Мирный** — хорошо спроектированный 2D top-down survival base-builder на Godot 4.6 с зрелой архитектурной базой. Проект перешёл из стадии прототипа в фазу активной разработки фундамента.

### Что реализовано и работает

Мировая генерация (чанковый стриминг, биомы, terrain, майнинг, топология гор, z-уровни) — около 85%. Билдинг-система (размещение, комнаты, indoor-солвер, персистентность) — 95%. Электроснабжение (supply/demand, приоритетный load-shedding) — 90%. Выживание (кислород indoor/outdoor, жизнеобеспечение) — 90%. Крафтинг (фреймворк рецептов) — 85% (но контента мало — 1 рецепт). Сохранение/загрузка (multi-file JSON, версия 4) — 95%. Локализация (ru/en фреймворк) — 85%. UI/HUD (инвентарь, строительное меню, крафт-панель, power UI) — 85%. Время/дневной цикл/сезоны — 95%. Команды (Command Pattern для всех мутаций) — 100%. State machine, компоненты, фабрики — всё на месте.

### Что ещё не реализовано или минимально

Контент (рецепты, флора, фауна, POI) — на уровне заглушек и плейсхолдеров. Аудио — 0%, нет никакой реализации. Продвинутый AI врагов — 30% (базовый слух, нет зрения/групп). Мультиплеер — 0%. GDExtension (нативный генератор чанков) — около 30%. Feature/POI система — MVP с тестовыми данными.

### Архитектурные сильные стороны

Проект имеет исключительно зрелую документацию для indie-игры. Документо-ориентированная модель управления (doc-driven governance) с чёткой иерархией приоритетов документов, обязательными контрактами данных, публичным API, стандартами инженерии и performance-контрактами. Это серьёзно снижает риск архитектурной деградации при работе с AI-ассистентами.

Паттерны (Command, Compute->Apply, Data-driven Registries, Deterministic Hashing, EventBus, Dirty Queue + Budget) реально используются в коде, а не просто задекларированы.

### Замечания

Единственный найденный TODO в коде: underground_fog_state.gd — радиус раскрытия захардкожен, нужно вынести в balance-ресурс. Есть несколько артефактных boot_fix логов в корне проекта (artifacts_boot_fix_v*.log), которые можно архивировать.

```
## Жёсткие правила работы

1. Документация прежде кода — не строй понимание архитектуры из кода, читай контракты и спеки.
2. Никакого широкого сканирования — не запускай поиск по всему репозиторию "для контекста". Если задача называет файлы — работай только с ними.
3. Одна задача — один шаг — не реализуй Iteration 2, если попросили Iteration 1. Не делай "заодно рефакторинг".
4. Минимальное валидное изменение — наименьшее изменение, удовлетворяющее спеке, контракту и acceptance-тестам.
5. Нет тихому дрифту контрактов — если код меняет data contract, owner boundary, API или инвариант, ОБНОВИ DATA_CONTRACTS.md и/или PUBLIC_API.md в той же задаче.
6. Стоп, когда готово — не продолжай из-за "соседних улучшений". Наблюдения вне скоупа → в секцию "Out-of-scope observations" closure report.

## Обязательные архитектурные паттерны

- Command Pattern — все мутации мира (build, mine, craft, pickup) через CommandExecutor.execute()
- Compute → Apply — тяжёлые операции: чистый compute (без scene tree) + bounded apply (main thread)
- Data-driven Registries — контент через .tres ресурсы с namespace:id, доступ через реестры, никогда load("res://...")
- Deterministic Hashing — вариация по позиции через pos.x * PRIME_A + pos.y * PRIME_B, никогда randf()/randi()
- EventBus — межсистемная коммуникация только через сигналы
- Dirty Queue + Budget — фоновая работа через FrameBudgetDispatcher (6ms/frame)

## Performance-контракты

- Интерактивные операции (mine, place, craft): < 2ms синхронно
- Фоновый бюджет: 6ms/frame (streaming > topology > visual > spawn)
- ЗАПРЕЩЕНО в интерактивном пути: full chunk redraw, full topology rebuild, mass add_child/queue_free/set_cell/clear, loop по всем загруженным чанкам
- Если один тайл изменился — обновляй только локальный dirty region

## Кодовые конвенции

- snake_case для файлов/переменных/функций, PascalCase для классов, UPPER_SNAKE_CASE для констант
- Сигналы: past tense (building_placed, item_crafted)
- Булевы: is_, has_, can_ префиксы
- Явная типизация на каждую переменную, параметр и возвращаемое значение
- Никакого пользовательского текста в коде — только Localization.t("KEY")
- Порядок в скрипте: class_name > extends > docs > signals > enums > constants > exports > public vars > private vars > Godot builtins > public methods > private methods
- Commit: feat(system):, fix(system):, refactor(system):, docs:, data(scope):
```

## Часть 3: Scheduled Tasks для Cowork

### Задача 1: Валидация целостности документации

**Расписание:** Раз в неделю (понедельник утром)

**Описание:** Проверяет, что ключевые документы проекта не содержат внутренних конфликтов.

**Промпт:**
```
Прочитай следующие документы проекта и проверь их на внутреннюю согласованность:

1. docs/00_governance/PUBLIC_API.md — проверь, что все safe entry points, перечисленные здесь, реально существуют в коде (проверяй grep-ом по точному имени функции)
2. docs/02_system_specs/world/DATA_CONTRACTS.md — проверь, что упомянутые слои данных и их владельцы соответствуют реальной структуре кода
3. docs/00_governance/SYSTEM_INVENTORY.md — проверь актуальность списка систем

Формат отчёта:
- Найденные расхождения между документами и кодом
- Функции из PUBLIC_API.md, которые не найдены в коде
- Системы из SYSTEM_INVENTORY.md, которых нет в коде (или наоборот)
- Рекомендации по обновлению

Сохрани отчёт как docs_integrity_report_ДАТА.md в корне проекта.
```

### Задача 2: Аудит performance-опасных паттернов

**Расписание:** Раз в неделю (среда)

**Описание:** Ищет в коде паттерны, нарушающие performance-контракты.

**Промпт:**
```
Проведи аудит performance-опасных паттернов в коде проекта, согласно docs/00_governance/PERFORMANCE_CONTRACTS.md.

Ищи:
1. Вызовы TileMapLayer.clear() в интерактивных путях
2. Циклы по всем загруженным чанкам (for chunk in get_loaded_chunks()) не в boot-time коде
3. Массовые add_child/queue_free в одном фрейме
4. Массовые set_cell в одном фрейме (вне boot)
5. Использование randf()/randi() для позиционно-зависимого контента (нарушение deterministic hashing)
6. Прямые вызовы приватных функций (_prefix) из PUBLIC_API.md forbidden-списков
7. Прямые load("res://data/...") вместо обращений к реестрам

Формат: файл, строка, описание проблемы, ссылка на нарушаемый контракт.
Сохрани отчёт как perf_audit_ДАТА.md в корне проекта.
```

**Промпт:**

```
1. Найди все пользовательские строки (в кавычках) в файлах scenes/ui/**/*.gd, которые не обёрнуты в Localization.t() или tr()

4. Найди неиспользуемые ключи в .po файлах

Сохрани отчёт как localization_audit_ДАТА.md в корне проекта.
```
